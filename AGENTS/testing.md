# Тесты в $mol / MAM

## Где живут

- Файлы **`*.test.ts`** рядом с модулем (или **`*.web.test.ts`** / **`*.node.test.ts`** только для платформы).
- MAM собирает **`web.test.js`** / **`node.test.js`** и **запускает тесты после сборки** (в Node — полный бандл приложения + тесты).

## API

- Обёртка **`$mol_test({ ... })`** — объект «имя сценария → функция».
- Функция получает **`$`** — контекст окружения (глобаль с модулями).
- Проверки: **`$mol_assert_equal`**, **`$mol_assert_ok`** и др. из **`$mol_assert`**.

Пример минимального теста:

```ts
namespace $ {

	$mol_test( {

		'my scenario'( $ ) {

			$mol_assert_equal( 1 + 1, 2 )

		},

	} )

}
```

Для **view-классов** часто делают `const x = new My_view(); x.$ = $`, чтобы подставить окружение (как в `mol/view/view/view.test.ts`).

## Запуск

1. **Сборка модуля** (подтянет и выполнит тесты):  
   `npm start eve/app`  
   или из корня MAM: **`npx mam eve/app`**.
2. **В браузере**: открыть **`/-/test.html`** у модуля — подгружается **`web.test.js`**, тесты гоняются при загрузке (удобно с breakpoints).
3. **Дев-сервер** `npm start`: при открытии приложения с **`index.html`** пересборка и перезагрузка; смотри лог консоли на падения тестов.

После сборки MAM запускает **`node eve/app/-/node.test.js`**: часть тестов из зависимостей (`$mol_state_local` и др.) пишет в **`~/.local/share/mol_state_local`**. Нужны права на домашний каталог; в sandbox/CI без этого шаг может упасть с `EACCES` — на обычной машине обычно ок.

## Зависимости между тестами

Сборщик **упорядочивает тесты по графу зависимостей модулей** (сначала нижние модули), чтобы при падении было проще локализовать причину.

## Практика для Eve

- Логику в **`*.view.ts`** (реактивные свойства, чистые ветки) выносить так, чтобы её можно было дернуть из **`$mol_test`** без полного DOM — как в **`eve/selection/multiple/multiple.test.ts`**.
- Для UI с DOM — создавать инстанс view, задавать **`.$`**, вызывать **`dom_tree()`** и проверять узел (паттерн из тестов **`$mol_view`**).
